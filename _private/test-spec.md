# test-spec

## 1. 目的

`_private/20260313/test-req.md` の要求を満たすため、`htmlparser` パッケージに対して `pytest` で実施する振る舞い中心のテスト仕様を定義する。

この仕様は、CLI 利用者から観測できる最終結果、出力、終了コード、および永続化データへの副作用を対象として、正常系と異常系のテスト観点とテストケースを明確にすることを目的とする。

## 2. スコープ

- 対象は `src/htmlparser/clix.py` が提供する `run`, `clear`, `count`, `print-list-text` の 4 コマンドとする。
- 主な対象ファイルは `src/htmlparser/clix.py`、`src/htmlparser/topconfigdb.py`、`pyproject.toml` とする。
- 設定ファイル読込、DB 初期化、DB 読込、DB 保存、件数表示、一覧表示、`run` の解析実行フローを利用者観点で検証することを対象とする。
- `pytest` による自動テストとして表現できることを対象とする。
- `Subapp` や各 scraper の HTML 解析アルゴリズム内部、`yklibpy` 内部実装、YAML ライブラリ内部挙動の詳細検証は対象外とする。

## 3. 参照ファイル

- `_private/20260313/test-req.md`
- `_private/fusion_class_top_and_clix-spec.md`
- `_private/refactoring_class_clix-spec.md`

- `src/htmlparser/topconfigdb.py`
- `src/htmlparser/subapp.py`

## 4. 現行外部仕様

### 4.1 CLI の現行公開挙動

- 公開サブコマンドは `run`, `clear`, `count`, `print-list-text` である。
- 全サブコマンドは必須引数 `config_file` を受け付ける。
- `print-list-text` はオプション引数 `--key` を受け付け、既定値は `"title"` である。
- 各コマンドは正常終了時に `0` を返す。
- `pyproject.toml` のエントリポイントは `htmlparser.clix:main` である。

### 4.2 コマンドごとの利用者観点の意味

- `run` は設定を読み込み、既存 DB を読み込んだうえで解析結果を反映し、保存後に件数を表示する。
- `clear` は DB データを消去して保存する。
- `count` は DB を読み込み、件数を表示する。
- `print-list-text` は DB を読み込み、指定キーの文字列一覧を表示する。

### 4.3 設定と初期化の利用者観点

- `config_file` には YAML の mapping を含む設定ファイルを与える。
- 設定には少なくとも `db_file` と `db_kind` が必要である。
- `db_kind` は DB 実装生成に使用され、非対応値は失敗として扱われる。
- `run` は必要に応じて `config_file` と `patterns` を用いて解析対象を決定する。

## 5. テスト仕様

### 5.1 基本方針

- テストは `pytest` で実装する。
- テストは内部メソッドの呼び出し順やローカル変数ではなく、利用者が観測できる結果を検証する。
- テストケースは、まず `正常系` と `異常系` に分け、その中で業務文脈または前提条件ごとにグルーピングする。
- テストケース名は、単体で仕様説明になる日本語文とする。
- 各テストケースは、少なくとも `事前条件`、`操作`、`期待結果`、`事後条件`、`不変条件` を明記する。

### 5.2 テストデータ方針

- 設定ファイル、DB ファイル、必要最小限の HTML 入力はテストごとに独立した一時領域へ配置する。
- DB の初期状態は、空 DB と既存データを持つ DB の両方を用意する。
- サンプル設定は、利用者向け意味が読み取れる最小構成を用いる。

サンプル設定ファイル例:

```yaml
db_kind: yaml
db_file: data.yaml
config_file: app.yaml
patterns:
  - sample-*.html
```

サンプル DB 初期データ例:

```yaml
item1:
  url: https://example.com/1
  title: title-1
item2:
  url: https://example.com/2
  title: title-2
```

### 5.3 正常系

#### 5.3.1 CLI 契約の維持

- テストケース名: `run, clear, count, print-list-text の4コマンドが継続して利用可能であること`
  - 事前条件: CLI を生成できること。
  - 操作: 利用可能なサブコマンド一覧を取得する。
  - 期待結果: `run`, `clear`, `count`, `print-list-text` が存在する。
  - 事後条件: CLI 定義が変更されていない。
  - 不変条件: 上記 4 個以外を必須追加しない。

- テストケース名: `全コマンドで config_file が必須引数として要求されること`
  - 事前条件: CLI を生成できること。
  - 操作: 各サブコマンドを `config_file` なしで解析する。
  - 期待結果: 引数不足として失敗する。
  - 事後条件: 任意既定値で補完されない。
  - 不変条件: `config_file` の必須性は変わらない。

- テストケース名: `print-list-text で key を省略したとき title を既定値として一覧表示すること`
  - 事前条件: `title` を含む DB データが存在する。
  - 操作: `print-list-text <config_file>` を実行する。
  - 期待結果: `title` の値一覧が表示される。
  - 事後条件: DB 内容は変更されない。
  - 不変条件: `--key` の既定値は `"title"` のままである。

#### 5.3.2 `count` の正常系

- テストケース名: `既存DBが2件あるとき count は件数を表示して0を返すこと`
  - 事前条件: 2 件のデータを持つ DB と、それを指す正しい設定ファイルが存在する。
  - 操作: `count <config_file>` を実行する。
  - 期待結果: 標準出力に `count=2` が表示され、終了コードが `0` である。
  - 事後条件: DB 内容は変更されない。
  - 不変条件: 件数表示の意味は DB エントリ件数である。

- テストケース名: `空DBのとき count は0件として表示して0を返すこと`
  - 事前条件: 空 DB と、それを指す正しい設定ファイルが存在する。
  - 操作: `count <config_file>` を実行する。
  - 期待結果: 標準出力に `count=0` が表示され、終了コードが `0` である。
  - 事後条件: DB は空のままである。
  - 不変条件: count 実行だけで保存内容は変化しない。

#### 5.3.3 `print-list-text` の正常系

- テストケース名: `title を持つDBに対して print-list-text はタイトル一覧を表示して0を返すこと`
  - 事前条件: 複数エントリが `title` を持つ DB が存在する。
  - 操作: `print-list-text <config_file>` を実行する。
  - 期待結果: 各エントリの `title` が 1 行ずつ表示され、終了コードが `0` である。
  - 事後条件: DB 内容は変更されない。
  - 不変条件: 表示対象は DB から取得した指定キーの文字列である。

- テストケース名: `key に url を指定したとき print-list-text はURL一覧を表示すること`
  - 事前条件: 複数エントリが `url` を持つ DB が存在する。
  - 操作: `print-list-text <config_file> --key url` を実行する。
  - 期待結果: 各エントリの `url` が 1 行ずつ表示される。
  - 事後条件: DB 内容は変更されない。
  - 不変条件: `--key` で指定した表示項目の意味は変わらない。

#### 5.3.4 `clear` の正常系

- テストケース名: `既存DBがあるとき clear は全データを消去して保存し0を返すこと`
  - 事前条件: 1 件以上のデータを持つ DB と、それを指す設定ファイルが存在する。
  - 操作: `clear <config_file>` を実行する。
  - 期待結果: 終了コードが `0` であり、再読込後の DB 件数が `0` である。
  - 事後条件: DB ファイルは空データとして保存されている。
  - 不変条件: コマンド名、引数仕様、保存形式は変わらない。

- テストケース名: `空DBに対して clear を実行しても空のまま0を返すこと`
  - 事前条件: 空 DB が存在する。
  - 操作: `clear <config_file>` を実行する。
  - 期待結果: 終了コードが `0` であり、DB は空のままである。
  - 事後条件: 不要なデータが追加されない。
  - 不変条件: clear は空 DB に対しても冪等である。

#### 5.3.5 `run` の正常系

- テストケース名: `既存DBと新規解析結果があるとき run は両者を反映して保存し件数を表示すること`
  - 事前条件: 既存 DB があり、解析対象設定と入力データが有効である。
  - 操作: `run <config_file>` を実行する。
  - 期待結果: 終了コードが `0` であり、保存後 DB に既存データと新規解析結果が反映され、標準出力に件数が表示される。
  - 事後条件: 解析結果を含む DB が保存される。
  - 不変条件: `run` は DB 読込、解析、保存、件数表示という利用者向け流れを維持する。

- テストケース名: `複数patternsがあるとき run は各patternを順に処理した結果をDBへ反映すること`
  - 事前条件: 複数の `patterns` を含む設定と、それぞれに対応する入力データが存在する。
  - 操作: `run <config_file>` を実行する。
  - 期待結果: 各 `pattern` に由来する結果が最終 DB に反映される。
  - 事後条件: 最終保存結果が全対象を含む。
  - 不変条件: `patterns` の複数指定で CLI 契約は変わらない。

- テストケース名: `既存DBが空でも run は解析結果だけを保存して件数を表示すること`
  - 事前条件: 空 DB と有効な解析入力が存在する。
  - 操作: `run <config_file>` を実行する。
  - 期待結果: 解析で得られた結果だけが保存され、件数が表示される。
  - 事後条件: DB は空ではなくなる。
  - 不変条件: 既存データの有無でコマンド契約は変わらない。

### 5.4 異常系

#### 5.4.1 `config_file` の前提違反

- テストケース名: `config_file が設定されていないとき処理を開始せず失敗すること`
  - 事前条件: `config_file` を持たない引数が渡される。
  - 操作: 対象コマンドを実行する。
  - 期待結果: `config_file is not set` 相当の失敗になる。
  - 事後条件: DB は作成も更新もされない。
  - 不変条件: 必須引数不足を黙って補完しない。

- テストケース名: `config_file が空文字のとき処理を開始せず失敗すること`
  - 事前条件: `config_file` に空文字が渡される。
  - 操作: 対象コマンドを実行する。
  - 期待結果: `config_file is empty` 相当の失敗になる。
  - 事後条件: DB は変更されない。
  - 不変条件: 空文字は有効パスとして扱わない。

- テストケース名: `設定YAMLがmappingでないとき設定不正として失敗すること`
  - 事前条件: YAML の内容が list や scalar である設定ファイルが存在する。
  - 操作: 対象コマンドを実行する。
  - 期待結果: `config file must contain a mapping` 相当の失敗になる。
  - 事後条件: DB は生成も更新もされない。
  - 不変条件: 設定ファイル形式は mapping 前提である。

#### 5.4.2 DB 初期化の前提違反

- テストケース名: `db_file が未設定のときDB初期化に失敗すること`
  - 事前条件: `db_file` を欠く設定ファイルが存在する。
  - 操作: 対象コマンドを実行する。
  - 期待結果: `db_file is not set` 相当の失敗になる。
  - 事後条件: DB は保存されない。
  - 不変条件: DB 保存先の暗黙補完はしない。

- テストケース名: `db_kind が未設定のときDB初期化に失敗すること`
  - 事前条件: `db_kind` を欠く設定ファイルが存在する。
  - 操作: 対象コマンドを実行する。
  - 期待結果: `db_kind is not set` 相当の失敗になる。
  - 事後条件: DB は保存されない。
  - 不変条件: DB 種別の暗黙補完はしない。

- テストケース名: `db_kind が非対応値のとき対応外として失敗すること`
  - 事前条件: `db_kind` に非対応値を設定した設定ファイルが存在する。
  - 操作: 対象コマンドを実行する。
  - 期待結果: `unsupported db_kind=...` 相当の失敗になる。
  - 事後条件: DB は生成されない。
  - 不変条件: 対応 DB 種別は明示的に判定される。

#### 5.4.3 `run` の前提違反

- テストケース名: `pattern の適用結果が無効なとき run は失敗して不正状態を保存しないこと`
  - 事前条件: `env.set_pattern()` が無効結果を返す入力条件を用意する。
  - 操作: `run <config_file>` を実行する。
  - 期待結果: `invalid result` 相当の失敗になる。
  - 事後条件: 不完全な解析結果で DB を保存しない。
  - 不変条件: 解析対象決定に失敗した場合は正常終了しない。

### 5.5 コメントと観測されたリスク

- `src/htmlparser/topconfigdb.py` は `config_file` が存在すると標準出力へデバッグ文字列を出力している。これは利用者向け仕様として安定させるべき出力には見えないため、テストでは必須期待値に含めず、不要出力として別途見直し候補とする。
- `run` は `get_env()` が `None` の場合に失敗ではなく処理スキップになる。`run` が何も解析しないまま正常終了しうるため、仕様として意図的か、実装上の見直し対象かを確認すべきである。
- `patterns` の既定値 `["Amazon-KU-1-file"]` は `TopConfigDb` 側の暗黙仕様であるが、上位仕様書では必須互換項目として十分に明文化されていない。現時点のテスト仕様では必須受け入れ条件にせず、補助的観測項目として扱う。
- YAML タグ付きロード互換は `count`、`print-list-text`、`run` の利用者観点の結果に影響するため、古い DB データ互換の回帰確認を将来追加してよい。

## 6. 非対象

- `Subapp` および各 scraper 内部の HTML 解析ロジック詳細の網羅検証
- `yklibpy` の `DbYaml`、`Env`、`UtilYaml` 自体の単体テスト
- 内部メソッドの呼び出し回数、ローカル変数、クラス配置などの実装詳細検証
- パフォーマンス測定、負荷試験、並列実行試験
- `run`, `clear`, `count`, `print-list-text` 以外の CLI 機能

## 7. 互換性方針

- 互換性維持の対象は、CLI 利用者から観測できるコマンド名、引数、終了コード、標準出力の意味、DB への副作用とする。
- `Top` 削除や `Clix` への責務集約のような内部構造変更は、利用者向け挙動が同じであれば許容する。
- テストは内部実装の再構成に追従しすぎず、公開契約の維持確認を優先する。

## 8. 実装制約

- テストコードは `pytest` で記述すること。
- テストは互いに独立し、ファイルシステム上の一時データを共有しないこと。
- 正常系・異常系の分類を崩さないこと。
- 各テストケース名は、仕様書を兼ねる日本語文で記述すること。
- 利用者から観測できない内部実装に直接依存しないこと。

## 9. 受け入れ条件

- `_private/test-spec.md` が `pytest` 前提のテスト仕様書として成立していること。
- 仕様書が `正常系` と `異常系` を最上位で分離していること。
- 正常系が少なくとも `run`, `clear`, `count`, `print-list-text` の公開挙動を網羅していること。
- 異常系が少なくとも `config_file` 不正、設定形式不正、`db_file` 未設定、`db_kind` 未設定、非対応 `db_kind`、`run` の前提違反を網羅していること。
- 各テストケースに `事前条件`、`操作`、`期待結果`、`事後条件`、`不変条件` が含まれていること。
- テストケース名だけでも期待される振る舞いを人間が理解できること。
- 仕様漏れ、仕様バグ、または実装疑義として観測された事項がコメントされていること。

## 10. 備考

- 本仕様書はテスト実装コードそのものではなく、振る舞いベースでテストを設計するための外部仕様書である。
- 本仕様書は現行の 4 コマンドを前提とし、旧仕様にある `prepare` は対象に含めない。
- 必要に応じて、本仕様書をもとに `tests/` 配下へ `pytest` テストコードを展開する。
